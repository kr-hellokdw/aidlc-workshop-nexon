# Unit 3: Admin Frontend - NFR Requirements (비기능 요구사항)

---

## 1. 보안 (Security)

### SEC-01: 인증 토큰 관리
| 항목 | 요구사항 |
|------|----------|
| **토큰 저장** | localStorage에 JWT 저장 (accessToken + refreshToken) |
| **토큰 전송** | 모든 API 요청에 `Authorization: Bearer <token>` 헤더 자동 첨부 |
| **토큰 갱신** | accessToken 만료 시 refreshToken으로 자동 갱신 (사용자 개입 없음) |
| **토큰 만료** | 16시간 후 자동 로그아웃 + 로그인 페이지 리다이렉트 |
| **토큰 삭제** | 로그아웃 시 localStorage에서 모든 인증 정보 즉시 삭제 |

### SEC-02: 입력 검증 (Client-side)
| 항목 | 요구사항 |
|------|----------|
| **XSS 방지** | 사용자 입력값 렌더링 시 React의 기본 이스케이핑 활용 (dangerouslySetInnerHTML 사용 금지) |
| **폼 검증** | 모든 폼 제출 전 클라이언트 측 유효성 검사 수행 |
| **파일 검증** | 이미지 업로드 시 클라이언트에서 형식(jpg/png/webp) + 크기(5MB) 사전 검증 |
| **URL 검증** | 외부 URL 입력 불허, API 응답의 imageUrl만 사용 |

### SEC-03: 인가 (Authorization)
| 항목 | 요구사항 |
|------|----------|
| **라우트 보호** | 인증되지 않은 사용자의 보호된 라우트 접근 차단 (AuthGuard) |
| **API 응답 처리** | 403 응답 시 적절한 에러 메시지 표시 (다른 매장 데이터 접근 시도 등) |
| **세션 무효화** | 서버에서 401 반환 + 토큰 갱신 실패 시 즉시 로그아웃 |

### SEC-04: 민감 정보 보호
| 항목 | 요구사항 |
|------|----------|
| **비밀번호** | 입력 필드 type="password", 자동완성 off |
| **환경변수** | API URL 등 설정값은 VITE_ 접두사 환경변수로 관리 |
| **소스코드** | 하드코딩된 비밀번호, API 키 등 민감 정보 포함 금지 |
| **에러 노출** | 서버 스택 트레이스, 내부 경로 등 사용자에게 노출 금지 |

## 2. 성능 (Performance)

### PERF-01: 초기 로딩
| 항목 | 목표 |
|------|------|
| **First Contentful Paint (FCP)** | 1.5초 이내 |
| **번들 크기** | 초기 로드 JS 번들 300KB 이하 (gzip 기준) |
| **코드 분할** | 라우트 기반 lazy loading (React.lazy + Suspense) |
| **정적 자산** | Vite 빌드 시 자동 해싱 + 브라우저 캐싱 활용 |

### PERF-02: 실시간 업데이트 (SSE)
| 항목 | 목표 |
|------|------|
| **이벤트 반영** | SSE 이벤트 수신 후 UI 반영 200ms 이내 |
| **연결 유지** | Heartbeat(30초) 기반 연결 상태 모니터링 |
| **재연결** | 연결 끊김 시 3초 후 재시도, 최대 5회 |
| **메모리** | SSE 이벤트 리스너 정리 (컴포넌트 unmount 시 구독 해제) |

### PERF-03: 렌더링 최적화
| 항목 | 목표 |
|------|------|
| **불필요한 리렌더링** | React.memo, useMemo, useCallback 적절히 활용 |
| **리스트 렌더링** | 대시보드 테이블 카드 (최대 20개) - key prop 올바르게 사용 |
| **이미지** | 메뉴 이미지 lazy loading (Intersection Observer 또는 loading="lazy") |
| **상태 업데이트** | SSE 이벤트 시 변경된 테이블 카드만 리렌더링 (불변 업데이트) |

### PERF-04: API 호출 최적화
| 항목 | 목표 |
|------|------|
| **중복 요청 방지** | 동일 요청 진행 중 추가 요청 차단 (debounce/throttle) |
| **낙관적 업데이트** | 주문 상태 변경 시 UI 즉시 반영 → 실패 시 롤백 |
| **에러 재시도** | 네트워크 오류 시 1회 자동 재시도 (5xx 에러) |

## 3. 사용성 (Usability)

### UX-01: 반응형 레이아웃
| 항목 | 요구사항 |
|------|----------|
| **최소 해상도** | 1024x768 (데스크톱/태블릿 가로 모드) |
| **대시보드 그리드** | 화면 너비에 따라 카드 열 수 자동 조정 (2~4열) |
| **사이드바** | 고정 너비 (240px), 축소 가능 |

### UX-02: 피드백
| 항목 | 요구사항 |
|------|----------|
| **로딩 상태** | 모든 API 호출 시 로딩 인디케이터 표시 |
| **성공 피드백** | Toast 알림 (3초 자동 닫힘) - 생성/수정/삭제 성공 |
| **실패 피드백** | Toast 알림 (수동 닫기) - 에러 메시지 표시 |
| **확인 다이얼로그** | 삭제, 이용 완료 등 되돌릴 수 없는 동작 전 확인 |
| **신규 주문 강조** | 2초간 테이블 카드 배경색 변경 (시각적 하이라이트) |

### UX-03: 네비게이션
| 항목 | 요구사항 |
|------|----------|
| **현재 위치** | 사이드바에서 현재 페이지 활성 상태 표시 |
| **페이지 전환** | 클라이언트 사이드 라우팅 (새로고침 없음) |
| **뒤로가기** | 브라우저 뒤로가기 정상 동작 |
| **SSE 상태** | 헤더에 연결 상태 아이콘 (초록: 연결됨, 빨강: 끊김) |

## 4. 접근성 (Accessibility)

### A11Y-01: WCAG 2.1 AA 준수
| 항목 | 요구사항 |
|------|----------|
| **색상 대비** | 텍스트/배경 대비 4.5:1 이상 (일반 텍스트), 3:1 이상 (큰 텍스트) |
| **터치 영역** | 모든 인터랙티브 요소 최소 44x44px |
| **포커스 표시** | 키보드 포커스 시 명확한 시각적 표시 (outline) |
| **포커스 순서** | 논리적 Tab 순서 (좌→우, 상→하) |

### A11Y-02: 스크린리더 지원
| 항목 | 요구사항 |
|------|----------|
| **시맨틱 HTML** | 적절한 heading 계층, landmark 역할 사용 |
| **폼 레이블** | 모든 입력 필드에 연결된 label 또는 aria-label |
| **에러 연결** | aria-describedby로 에러 메시지와 입력 필드 연결 |
| **동적 알림** | 주문 상태 변경, 토스트 등 aria-live="polite" 영역 사용 |
| **모달** | aria-modal="true", focus trap, ESC 닫기 |

### A11Y-03: 키보드 접근성
| 항목 | 요구사항 |
|------|----------|
| **전체 기능** | 마우스 없이 키보드만으로 모든 기능 사용 가능 |
| **드래그 대안** | 메뉴 순서 변경 시 키보드 대안 제공 (위/아래 버튼) |
| **단축키** | ESC: 모달 닫기, Enter: 폼 제출 |

## 5. 신뢰성 (Reliability)

### REL-01: 에러 복구
| 항목 | 요구사항 |
|------|----------|
| **Error Boundary** | 컴포넌트 에러 시 전체 앱 크래시 방지, 에러 UI 표시 |
| **SSE 재연결** | 연결 끊김 시 자동 재연결 (3초 간격, 최대 5회) |

| **네트워크 복구** | 네트워크 복구 시 대시보드 데이터 자동 새로고침 |

### REL-02: 데이터 일관성
| 항목 | 요구사항 |
|------|----------|
| **SSE 동기화** | SSE 재연결 시 REST API로 전체 상태 재조회 (누락 이벤트 보상) |
| **낙관적 업데이트** | 서버 응답과 로컬 상태 불일치 시 서버 데이터 우선 |
| **동시 수정** | 같은 주문 동시 상태 변경 시 서버 최종 상태 반영 |
