## 5. API 연동 규칙

### 사용할 Backend API 목록

| API | 메서드 | 용도 | 인증 |
|-----|--------|------|------|
| `/api/auth/table/login` | POST | 태블릿 로그인 | 불필요 |
| `/api/stores/{storeId}/categories` | GET | 카테고리 목록 | TABLE 토큰 |
| `/api/stores/{storeId}/menus` | GET | 메뉴 목록 | TABLE 토큰 |
| `/api/orders` | POST | 주문 생성 | TABLE 토큰 |
| `/api/orders/current` | GET | 현재 세션 주문 | TABLE 토큰 |
| `/api/files/images/{filename}` | GET | 메뉴 이미지 | 불필요 |

### 에러 처리 매핑

| HTTP 상태 | 사용자 메시지 |
|-----------|---------------|
| 400 | "입력 정보를 확인해주세요" |
| 401 | (자동 재로그인 시도, 실패 시 설정 화면) |
| 404 | "요청한 정보를 찾을 수 없습니다" |
| 500 | "서버 오류가 발생했습니다. 잠시 후 다시 시도해주세요" |
| 네트워크 에러 | "네트워크 연결을 확인해주세요" |

## 6. Mock API 구조 (독립 개발용)

### MSW 핸들러 구성
```
src/mocks/
├── handlers.ts       # 전체 핸들러 등록
├── data.ts           # Mock 데이터 (카테고리, 메뉴, 주문)
├── auth.handlers.ts  # POST /api/auth/table/login
├── menu.handlers.ts  # GET categories, GET menus
├── order.handlers.ts # POST orders, GET orders/current
└── browser.ts        # MSW 브라우저 워커 설정
```

### Mock 데이터 예시
- 카테고리 3~5개 (한식, 음료, 사이드 등)
- 메뉴 10~15개 (카테고리별 3~5개)
- 주문 2~3개 (다양한 상태)
